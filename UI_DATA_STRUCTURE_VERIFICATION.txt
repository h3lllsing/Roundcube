F. VPS SPECIAL VERIFICATION — MULTI-IP SUPPORT
------------------------------------------------
  Q: Can one VPS already store multiple IPs?
  A: NO. `ip_address` is a single string (max:45). One IP only.

  Q: Can one VPS already store multiple login IDs?
  A: NO. No field exists for login IDs at all.

  Q: Can one VPS already store multiple passwords?
  A: NO. `password` is a single encrypted text field.

  Q: Is there already JSON support?
  A: NO. No JSON columns on VPS model. (But `extensions` on VoIP is JSON.)

  Q: Is there already notes/details field that could store multiple values?
  A: The `notes` field is a text column. It's unstructured — could store
     anything but has no JSON casting. Would require structural parsing.

  Q: Would supporting multiple IPs require:
      a) UI change only — NO. Needs data model change.
      b) Model change — YES. Need JSON cast or new relationship.
      c) Database change — YES. Need JSON column or pivot table.
      d) New table — POSSIBLE. Could use a new `vps_ips` table or
         add a `json` column `ip_addresses`. The VoIP `extensions` pattern
         (JSON array on model) is precedent for option (b)+(c).

7. MODULE FIELD VERIFICATION
================================================================================

A. WHAT TABLE IT COMES FROM
----------------------------
The `module_id` column on each resource table is a FK to `modules`
table. The Module model lives at app/Models/Module.php. The modules
table has: id, feature_id (FK → features), name, slug, description,
is_active, created_by, updated_by, timestamps, softDeletes.

The `module` relationship on each resource model returns the related
Module via BelongsTo(Module::class).

B. WHY IT EXISTS
-----------------
Modules represent logical groupings of resources within features.
Examples: "Domains" module under "Infrastructure" feature, "VPS" module
under "Infrastructure" feature. They enable role-based access control
(RBAC) at a granular level — a user can be given read/write/export
permissions on specific modules but not others.

C. WHAT FUNCTIONALITY DEPENDS ON IT
-------------------------------------
- Role-based permissions: ModuleRolePermission table ties roles to
  modules with can_read/can_write/can_export flags. Resource controllers
  check `canOnModule()` before allowing access.
- Scoping: Non-super-admin users see only resources in their accessible
  modules (via `accessible_module_ids` in services/controllers).
- Dashboard/API filtering: Both Web and API controllers allow filtering
  by `module_id`.
- Resource assignment: Create/Edit forms include a `module_id` select.

D. WHETHER PERMISSIONS DEPEND ON IT
-------------------------------------
YES. Module permissions are the primary access control mechanism for
non-super-admin users. The `canOnModule()` method (from
HasModulePermissions trait) checks:
  1. If user is super-admin → always true
  2. If module has no role permissions → true
  3. Otherwise checks ModuleRolePermission for user's role + module

This is used in: VpsController, HostingController, ServiceProviderController,
DomainEmailController, VoipController, DomainController, TaskController,
ExportController (Web + API), and DashboardController.

E. WHETHER SEARCH DEPENDS ON IT
---------------------------------
YES. The API SearchController and all service classes (VpsService,
HostingService, etc.) filter results by `accessible_module_ids` for
non-super-admin users. This ensures users only see resources from
modules they have permission to access.

F. WHETHER EXPORTS/IMPORTS DEPEND ON IT
-----------------------------------------
YES. ExportController (Web + API) checks module_slug permission via
`canOnModule()` with the 'export' ability if the user is not super-admin
and the type has a `module_slug` configured. ImportController does NOT
currently check module permissions but sets user_id from Auth::id().

G. WHETHER REMOVING IT WOULD BREAK ANYTHING
---------------------------------------------
REMOVING THE MODULE FIELD WOULD BREAK:
  - All role-based access control for resources
  - Dashboard module filtering
  - API endpoint module_id filters
  - Search scoping
  - Export permission checks
  - The entire ModuleRolePermission system
  - Feature → Module hierarchy navigation
  - Create/Edit forms (module_id dropdown removal)

The module field is architecturally fundamental. Removing it would
require a full permission system redesign and is NOT recommended.
