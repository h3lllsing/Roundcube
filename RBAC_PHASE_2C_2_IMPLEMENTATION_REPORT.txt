RBAC Phase 2C-2 Implementation Report
=======================================
Date: 2026-06-25
Scope: Edit/Delete button visibility on index + show pages (9 resources)

Summary
-------
Phase 2C-2 hides/shows Edit and Delete per-record action buttons based on
`can_update` / `can_delete` permissions.  All changes are view-layer only;
server-side guards (Phase 2B-2) remain authoritative.

Changes Made
------------

### 1. Controllers — add `module_id` to `select()` (9 files)

Each controller's `index()` method used `->select(...)` to limit columns, but
none included `module_id`.  Without it, the eager-loaded `module` relationship
never resolved (`$record->module` was always null), so per-row permission checks
in Blade would short-circuit to false and hide buttons even when the user had
permission.

Pattern: added `'module_id'` to the `select([...])` array.

### 2. Index views — Edit/Delete per-row actions (9 files)

Wrapped the Edit and Delete `<x-action>` components in `@if` conditions:

  @if(auth()->user()->hasRole('super-admin') || ($record->module &&
      auth()->user()->canOnModule($record->module, 'update')))
      <x-action ... icon="edit" ... />
  @endif

  @if(auth()->user()->hasRole('super-admin') || ($record->module &&
      auth()->user()->canOnModule($record->module, 'delete')))
      <x-action ... icon="delete" ... />
  @endif

*View* button remains unconditional (read-permission already enforced by
RbacScope).

### 4. Tests (new file)

  tests/Feature/RbacPhase2C2Test.php — 18 tests

Tests cover:
  - Super-admin sees Edit/Delete on index + show
  - Edit visible when can_update=true, hidden when false
  - Delete visible when can_delete=true, hidden when false
  - User overrides (override=true shows, override=false hides)
  - Server-side guards still return 403 for direct URL access
  - Coverage across domains, hostings, vps, vault, other-services

Note: `assertSeeHtml`/`assertDontSeeHtml` used for delete assertions because
`assertSee` escapes HTML characters (turning `"` into `&quot;`), which would
cause false positives/negatives when matching `action="..."` attributes.

Design Decisions
----------------
- Permission logic computed inline in Blade (no controller changes for Phase
  2C-2) because `canOnModule()` is per-record and depends on `$record->module`
  which is already eager-loaded via `->with('module')`.
- `module_id` added to `select()` in all 9 controllers. Without it the
  eager-loaded module relationship silently fails to match (Laravel needs the
  foreign key in the result set).
- Super-admin shortcut (`hasRole('super-admin')`) checked before the
  canOnModule call so SA always sees all buttons regardless of DB permissions.
- User overrides work automatically through `canOnModule()` — no additional
  logic needed.
