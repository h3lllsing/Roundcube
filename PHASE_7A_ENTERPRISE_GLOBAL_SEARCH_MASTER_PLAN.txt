1. CURRENT STATE ANALYSIS
=======================================================================

  Existing SearchController (app/Http/Controllers/Web/SearchController.php):
    - 16 module types in a hardcoded $searches array
    - Single-column LIKE search on each model's primary display column
    - Non-SA scoping: user_id filter for "user owned types", module_id IN for tasks
    - No multi-column search (except Assets which also searches serial_number)
    - No search result highlighting
    - No recent searches
    - No per-module result limits
    - No keyboard-friendly result navigation on search page
    - No pagination on the search results page
    - View (search/index.blade.php) groups results by type in cards

  Command Palette (admin.blade.js, inline JS):
    - Hardcoded client-side cmdPages array (page labels + URLs only)
    - Filtered by label match only — no record-level search
    - Super-admin entries gated by @hasrole blade directive
    - NOT connected to SearchController
    - No API endpoint for live search results

  Sidebar Search:
    - Form submits GET to /search?q=...
    - Same SearchController as above
    - Placeholder says "Search... (Ctrl+K)" but Ctrl+K opens cmd palette (different)

  Gaps:
    - Only searches primary column per model (e.g. name only, not email or IP)
    - No search across secondary identifiers (cPanel ID, asset serial, hostname, IP)
    - No results inside Ctrl+K modal
    - No recent searches stored
    - No text highlighting
    - No per-module result caps
    - Admin role not properly scoped (no use of getAccessibleModuleIds for most types)

2. ARCHITECTURE OVERVIEW
=======================================================================

  Global Search Service
    App\Services\GlobalSearchService
    - Single class with search() method
    - Accepts: string $query, User $user, ?int $perModuleLimit = 5
    - Returns: Collection of SearchResultGroup objects

  SearchResultGroup DTO
    App\DTOs\SearchResultGroup
    - string $moduleKey (slug)
    - string $label (display name)
    - string $route (index route for "View all")
    - Collection $items (SearchResultItem objects)

  SearchResultItem DTO
    App\DTOs\SearchResultItem
    - mixed $id
    - string $title (highlighted)
    - ?string $subtitle (highlighted)
    - string $url (show route)
    - ?string $badge (status/label)
    - ?string $category (category label)

  Highlight helper
    App\Helpers\SearchHelper::highlight(string $text, string $query): string
    - Wraps matching portions in <mark> tags
    - Case-insensitive matching

  API Endpoint (for Ctrl+K modal)
    GET /api/search?q=...
    Returns JSON: { groups: [{ key, label, route, items: [{ id, title, subtitle, url, badge }] }] }

  Web Endpoint (full search page)
    GET /search?q=...
    Existing route, refactored to use GlobalSearchService
    Pagination via ?page=N (for "View all" per-module drill-down)

  Recent Searches
    - Stored in database: user_search_history table
    - Columns: id, user_id, query, created_at
    - Migration: create_user_search_history_table
    - Max 10 recent searches per user
    - Trimmed on insert
    - No migration = store in localstorage as fallback

3. SEARCHABLE MODULES — PER-MODULE SPEC
=======================================================================

  3.1 Clients
  -----------
  Note: No Client model exists. Clients are Users in this system.
  Plan: Search the users table as "Clients" for non-SA roles that
        need to find user records without full user management.

  Model:    User
  Table:    users
  Columns:  name, email
  Title:    name
  Subtitle: email
  Badge:    User's primary role slug (via Tyro roles)
  URL:      route('users.show', $user->id)
  Permission: super-admin only (users.show is under role:super-admin middleware)
  Ownership:  super-admin sees all; admin/user sees nothing
  Limit:     5
  Notes:     Search both name and email via OR WHERE LIKE

  3.2 Domains
  -----------
  Model:    App\Models\Domain
  Table:    domains
  Columns:  name, dns_servers (JSON array — cast to string in search), notes
  Title:    name
  Subtitle: serviceProvider?->name (provider name)
  Badge:    status
  URL:      route('domains.show', $domain->id)
  Permission: 'domains' module — can_read check via getAccessibleModuleIds('read')
             + user_id ownership fallback for regular users
  Ownership:
    - Super-admin: all domains
    - Admin: domains where module_id IN accessibleModuleIds('read')
    - Regular user: domains where user_id = $user->id
  Limit:     5
  Notes:     Index name column. dns_servers is JSON, difficult to LIKE search
             in SQLite. Skip dns_servers in initial implementation.
             Search name with LIKE, optionally search notes.

  3.3 Hosting
  -----------
  Model:    App\Models\Hosting
  Table:    hostings
  Columns:  name, username, domain, cpanel_ip (added 2026_06_25_000002),
            domain_ip (added 2026_06_25_000002), notes
  Title:    name
  Subtitle: plan ?? domain
  Badge:    status
  URL:      route('hostings.show', $hosting->id)
  Permission: same as Domains (module slug: 'hostings')
  Ownership:
    - Super-admin: all
    - Admin: module_id IN accessibleModuleIds('read')
    - Regular user: user_id = $user->id
  Limit:     5
  Notes:     Search: name LIKE, username LIKE, domain LIKE, cpanel_ip LIKE.
             Do NOT expose password column.

  3.4 VPS

  Columns:  name, ip_address, additional_ips (JSON array), os,

  Subtitle: ip_address ?? os
  Badge:    status
  URL:      route('vps.show', $vps->id)
  Permission: same pattern as Domains (module slug: 'vps')
  Limit:     5
  Notes:     Search: name LIKE, ip_address LIKE.
             additional_ips is JSON, hard to search — skip initially.
             Search os, department, location as supplementary columns.

  3.5 Domain Emails

  Title:    email

  Permission: same pattern as Domains (module slug: 'domain-emails')
  Limit:     5

  3.10 Tasks
  ----------
  Model:    App\Models\Task
  Table:    tasks
  Columns:  title, description
  Title:    title

  Permission: Tasks are NOT module-scoped like services.

    - Admin: tasks where module_id IN accessibleModuleIds('read')

    - Regular user: tasks where user is assignee
  Limit:     5
  Notes:     Search: title LIKE, description LIKE.
             Eager load assignees for subtitle (optional).

  3.11 Vault / Credentials
  ------------------------
  Model:    App\Models\VaultEntry
  Table:    password_vault
  Columns:  service_name, service_url, username, description

  URL:      route('vault.show', $vault->id)

            Only surface service_name, username, service_url in results.

4. PERMISSION & OWNERSHIP MODEL
=======================================================================

  Permission Matrix:

  Module              SA      Admin           IT Support       User/Read Only
  ─────────────────────────────────────────────────────────────────────────────
  Clients/Users       ALL     —               —                —
  Domains             ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  Hosting             ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  VPS                 ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  Domain Emails       ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  VoIP                ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  Other Services      ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  Expiry Trackers     ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds
  Assets              ALL     module_id IN     module_id IN     user_id = me
                              accessibleIds    accessibleIds  OR assigned_to = me
  Tasks               ALL     module_id IN     module_id IN     assigned to me
                              accessibleIds    accessibleIds
                   OR assignee = me       OR assignee = me
  Vault               ALL     user_id = me    user_id = me     user_id = me
  Notes               ALL     user_id = me    user_id = me     user_id = me
  SMTP Profiles       ALL     —               —                —
  Activity Logs       ALL     —               —                —

  Implementation approach:
    GlobalSearchService receives User, resolves accessibleModuleIds upfront.
    Each module search method receives the same accessibleIds + User.
    Ownership scoping uses the SAME logic as RbacScope::apply() from app/Helpers.

6. API & CONTROLLER DESIGN
=======================================================================

  6.1 GlobalSearchService
  -----------------------
    Namespace: App\Services\GlobalSearchService
    Method: search(string $query, User $user, int $perModuleLimit = 5): array
    Returns: array of SearchResultGroup (or DTO collections)

    Internal structure:
      - Resolves accessibleModuleIds via $user->getAccessibleModuleIds('read')
      - Defines $modules config (key -> [model, columns, title_col, subtitle_col,
        badge_col, route_name, permission_callback, scope_callback])
      - Iterates modules, builds query with LIKE on each column
      - Applies ownership/permission scoping
      - Limits per module, constructs SearchResultItem DTOs
      - Applies highlighting
      - Returns grouped results

    Search logic per module:
      $query->where(function($q) use ($term, $columns) {
          foreach ($columns as $col) {
              $q->orWhere($col, 'like', $term);
          }
      });

7. CACHE & PERFORMANCE STRATEGY
=======================================================================

   7.1 Per-User Query Cache
   ------------------------
     Key: "search:{$user->id}:".md5($query)
     TTL: 60 seconds (matching existing API cache)
     Store: Cache::remember (default driver — file/database/redis)

   7.2 Query Optimization
   ----------------------
     - Select ONLY needed columns (id, title_col, subtitle_col, badge_col,
       route-related FK)
     - Use ->take($perModuleLimit) on each subquery
     - Eager load only used relations (eager load via with(), not lazy)
     - No eager loading of polymorphic relations in search results
       (use stored column values instead)

   7.3 Index Recommendations (future migration)
   --------------------------------------------
     - domains.name -> INDEX
     - hostings.name -> INDEX
     - vps.name, vps.ip_address -> INDEX
     - assets.asset_tag (unique, already), assets.serial_number -> INDEX
     - expiry_trackers.name -> INDEX
     - tasks.title -> INDEX
     - password_vault.service_name -> INDEX
     - users.name, users.email -> INDEX
     - notes.content -> FULLTEXT (MySQL 5.7+ only)
     - activity_log.description -> FULLTEXT (MySQL 5.7+ only)

     Without these indexes, LIKE '%term%' on large tables will be slow.
     Initial implementation uses LIKE with LIMIT; fulltext is documented
     as a future optimization.

   7.4 Avoiding N+1
   -----------------
     All queries use ->take(5) to limit results.
     Eager load only when essential for subtitle/display.
     For route URLs, use route() helper with the ID — no need to load model.

   7.5 Shared-Hosting Compatibility
   ---------------------------------
     - No Redis, no Elasticsearch, no Meilisearch, no Algolia
     - LIKE search works on SQLite, MySQL, MariaDB, PostgreSQL
     - All caching uses default Laravel cache driver
     - No raw SQL that varies by database driver
     - Thread-safe: no file-based race conditions

11. FUTURE CONSIDERATIONS (DOCUMENTED ONLY — NOT IMPLEMENTED)
=======================================================================

  11.1 Full-Text Search
       MySQL / MariaDB: ALTER TABLE ... ADD FULLTEXT INDEX on text columns
       PostgreSQL: Use tsvector/tsquery via Laravel Scout
       Requires: Database-specific migration + conditional query logic
       Benefit: 10-100x faster than LIKE '%term%' on large datasets
       Risk: Not shared-hosting compatible; MySQL 5.7+ required

  11.2 Search Synonyms
       Map common terms: "vps" -> "server", "email" -> "mail", "host" -> "hosting"
       Store in config/search-synonyms.php
       Apply before query: expand term to include synonyms

  11.3 Fuzzy Search
       Allow typos: "exmaple" -> "example"
       Implement via: Levenshtein distance (MySQL), pg_trgm (PostgreSQL),
       or client-side suggestion
       NOT for initial implementation — adds complexity and performance cost

  11.4 Search Relevance Scoring
       Weight columns: title 10x, subtitle 3x, other 1x
       Order results by relevance score
       Currently: grouped by module, limited per module — relevance across
       modules is less important than module grouping

  11.5 Saved Searches
       Allow users to save named searches for quick access
       Different from recent searches — persisted indefinitely
       Could be stored in the same user_search_history table with
       an is_saved flag

  11.6 Autocomplete Suggestions
       Show suggestions as user types (before pressing Enter)
       Based on: recent searches, popular searches (if tracked),
       matching terms from database
