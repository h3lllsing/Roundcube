BETTER CREATE USER — ARCHITECTURE REVIEW
=========================================
Date: 2026-06-25
Reviewing: PHASE_3A3_BETTER_CREATE_USER_PLAN.txt

================================================================================
VERIFICATION OF CONCERN
================================================================================

The concern is VALID and represents a critical architectural flaw.

Tracing the exact failure scenario from the current plan (Step 2):

  1. Role "IT Support" exists with ModuleRolePermission for 6 modules.
     10 users currently have this role assigned.

  2. Super Admin opens Create User form.
     Selects role "IT Support" checkbox in Section 1.
     Selects "Admin" template in Section 2 Quick Setup.

  3. store() fires Step 2:
     $template->applyTemplateToRole($role) is called for role = IT Support.
     This calls ModuleRolePermission::updateOrCreate(module_id, role_id, ...)
     for every module slug defined in the Admin template's permissions_json.

  4. RESULT: The IT Support role's ModuleRolePermission rows are OVERWRITTEN
     to match the Admin template values (infrastructure R/C/U/E/Rev instead of
     just R/C/U/Rev; productivity R/C/U added; etc.).

  5. CONSEQUENCE: ALL 10 existing IT Support users immediately inherit the new
     Admin-level permissions. Their effective permissions change without notice,
     without their knowledge, and without any explicit role management action.

  6. AUDIT FAILURE: The activity log shows 'template_applied_during_creation'
     with the new user as $subject, making it appear to be a user-specific
     operation when in fact it mutated a shared resource.

This is not a hypothetical edge case — it is the direct consequence of Step 2
in the current plan. The architecture violates the fundamental RBAC separation
between role-level configuration and user-level assignment.

================================================================================
QUESTION 1: Should Role Templates ONLY be applied while creating/editing Roles?
================================================================================

YES.

Role Templates modify ModuleRolePermission, which is inherently a role-scoped
resource. Every user who holds that role is affected. Therefore the only correct
place to apply a template is when configuring the role itself:

  Role Templates page         →    Role Index/Edit page
  (template acts as preset)        (modifies ModuleRolePermission)

The Role Templates feature (Phase 3A-2) already implements this correctly:
Super Admin navigates to a template, selects a target role, previews the diff,
confirms, and the template is applied to that role's permissions. All affected
users see the change only when their role is explicitly reconfigured.

Applying a template during user creation would bypass the diff preview, bypass
the confirmation for affected existing users, and mix two separate concerns.

================================================================================
QUESTION 2: Should User creation simply assign/roles/clone/override?
================================================================================

YES. User creation should operate exclusively on user-scoped resources:

  1. CREATE the user record (name, email, password, status).
  2. ASSIGN existing roles via the user_role pivot table.
     These roles already have their permissions configured.
     No ModuleRolePermission rows are created or modified.
  3. OPTIONALLY CLONE another user's roles and overrides.
     This copies pivot rows and UserModulePermission rows only.
     No ModuleRolePermission rows are created or modified.
  4. OPTIONALLY SET user overrides via UserModulePermission.
     This is the only user-scoped permission mechanism.
     Overrides take precedence per RBAC v1.0 freeze rule.

The controller should ONLY write to:
  - users table (the new user record)
  - user_role pivot table (role assignments)
  - user_module_permissions table (user overrides)

It should NEVER write to:
  - module_role_permissions table (role-scoped, shared)

================================================================================
QUESTION 3: Would this architecture be cleaner?
================================================================================

YES. The architecture becomes three strict layers with one-directional influence:

  LAYER 1: ROLE TEMPLATES
    Resource: role_template + role_template_modules (JSON)
    Writes to: module_role_permissions
    Scope: role-level — affects all users with that role
    Entry point: Role Templates UI (index → show → apply)

  LAYER 2: ROLE ASSIGNMENT
    Resource: user_role pivot table
    Writes to: user_role pivot only
    Scope: user-level — affects only the specific user
    Entry point: User Create / User Edit UI

  LAYER 3: USER OVERRIDES
    Resource: user_module_permissions
    Writes to: user_module_permissions only
    Scope: user-level — affects only the specific user
    Entry point: User Create / User Edit UI (collapsible)

  DATA FLOW:
    Templates ──[apply]──> Roles ──[assign]──> Users ──[override]──> Overrides
       ^                      ^                    ^                      |
       |                      |                    |                      |
    Role Templates          RoleMgmt             UserMgmt           UserMgmt
       UI                     UI                   UI                    UI

Each layer is independently auditable, independently testable, and independently
modifiable. No layer writes to a resource owned by another layer.

================================================================================
QUESTION 4: Would this reduce accidental permission changes?
================================================================================

YES. Significantly.

With the current plan, a single user creation could accidentally:

  - Grant can_delete to all IT Support users (if Admin template applied to
    IT Support role — Admin template has can_delete=false, but even changing
    can_read from false to true on a module would affect all)

  - Export or reveal permissions suddenly appear for users who never had them

  - A misclick on the template dropdown immediately affects production users
    with no intermediate confirmation about the role-wide impact

With the corrected architecture:

  - User creation only affects the single user being created
  - Role permissions are only modified through the Role Templates UI which:
    a) Shows a diff preview of exactly what will change and which role
    b) Requires explicit confirmation
    c) Logs the change as a role-scoped event (not a user-scoped event)
  - The blast radius of any single operation is minimized

================================================================================
QUESTION 5: Would this better follow RBAC principles?
================================================================================

YES. The corrected architecture directly implements NIST RBAC (ANSI INCITS 359):

  Core RBAC:
    - Users are assigned to Roles (user_role pivot) ✓
    - Roles have Permissions (module_role_permissions) ✓
    - Users acquire Permissions through Role membership ✓

  The current plan violates this by making User creation also modify Role
  permissions, effectively creating a hidden side-channel where a user-level
  operation changes role-level resources.

  The corrected architecture maintains strict separation:
    - Role Templates → Role Permission Configuration
    - User Management → User-to-Role Assignment + User Overrides

================================================================================
QUESTION 6: Would this simplify auditing?
================================================================================

YES. The audit log becomes self-documenting:

  Current plan audit entries (confusing):
    event: 'template_applied_during_creation'
    subject: User (newly created user)
    properties: { template: 'Admin', role_count: 2 }
    → MISLEADING: implies this only affected the new user
    → REALITY: affected ALL users of those 2 roles

  Corrected architecture audit entries (clear):
    # Role management event (separate, role-scoped)
    event: 'template_applied'
    subject: Role (IT Support)
    properties: { template: 'Admin', changed_count: 15 }
    → CLEAR: this changed the IT Support role; all 10 members affected

    # User management event (separate, user-scoped)
    event: 'created'
    subject: User (new user)
    properties: { roles: ['IT Support'] }
    → CLEAR: this only created a user and assigned them to IT Support

  Auditors can trace:
    1. "When was the IT Support role last modified?" → template_applied event
    2. "Which users have the IT Support role?" → user_role pivot query
    3. "Does user X have overrides?" → user_module_permissions query

  Each question maps to exactly one event type and one resource.

QUESTION 7: Would this reduce risk in production?
================================================================================

YES. Quantifiable risk reduction:

  RISK REMOVED: Silent role-wide permission change during user creation
    Severity: High (affects all users of a role, no confirmation, no warning)
    Mitigated by: Removing template application from user creation entirely

  RISK REMOVED: Audit trail obscuring role-wide changes as user-level events
    Severity: Medium (complicates incident response, hides root cause)
    Mitigated by: Only logging user-scoped events during user creation

  RISK REMOVED: Accidental Super Admin template applied to shared role
    Severity: Critical (grants full access to all role members)
    Mitigated by: Super Admin template can only be applied through
    Role Templates UI with diff preview + dangerous confirmation

  RISK REDUCED: Form complexity
    Severity: Low (UI confusion, potential misclicks)
    Mitigated by: Removing Section 2 Quick Setup entirely simplifies the form

  The corrected architecture eliminates the highest-severity risk (silent
  role-wide permission mutation) completely.

REVISED CONTROLLER CHANGES
==========================

  create():
    - Load roles (existing)
    - Load modules with features (existing)
    - Pre-compute role summaries (NEW — role name + template slug + module
      info, for the informational preview)
    - Load users for clone search (NEW)

  store():
    - Validate basic fields (existing + status)
    - Validate clone fields (NEW: clone_user_id, copy_roles, etc.)
    - Create user (existing)
    - Handle status (NEW: suspended_at based on status dropdown)
    - Handle clone (NEW: copy roles/overrides/status from source)
    - Sync manually selected roles (existing)
    - Handle special overrides (existing saveUserModulePermissions)
    - Do NOT apply any template (REMOVED)
    - Do NOT write to module_role_permissions (REMOVED)

  store() pseudocode:
    function store(Request $request) {
        $validated = $request->validate([...existing fields..., 'status', ...clone fields...]);

        DB::transaction(function () use ($request, $validated) {
            // 1. Create user
            $user = User::create([...]);

            // 2. Handle status
            if ($request->input('status') === 'suspended') {
                $user->suspended_at = now();
                $user->save();
            }

            // 3. Handle clone
            if ($request->filled('clone_user_id')) {
                $sourceUser = User::findOrFail($request->input('clone_user_id'));

                if ($request->boolean('copy_roles')) {
                    $roleIds = $sourceUser->roles->pluck('id')->toArray();
                    $this->preventSuperAdminAssignment($roleIds);
                    $user->roles()->syncWithoutDetaching($roleIds);
                }

                if ($request->boolean('copy_overrides')) {
                    // Copy UserModulePermission rows from source
                }

                if ($request->boolean('copy_status') && $sourceUser->suspended_at) {
                    $user->suspended_at = $sourceUser->suspended_at;
                    $user->save();
                }
            }

            // 4. Sync manually selected roles
            if ($request->has('roles')) {
                $user->roles()->syncWithoutDetaching($request->input('roles', []));
            }

            // 5. Handle overrides (existing)
            $this->saveUserModulePermissions($user, $request);
        });

        return redirect()->route('users.index')
            ->with('success', 'User created successfully.');
    }

  The controller NEVER calls applyTemplateToRole, NEVER imports
  RoleTemplateController, and NEVER writes to module_role_permissions.

QUESTION 10: RECOMMENDED FINAL ENTERPRISE ARCHITECTURE
================================================================================

The final recommended architecture for user + role management in this system:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                    SEPARATION OF CONCERNS                           │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  ROLE CONFIGURATION (Role Templates)                                │
  │  ───────────────────────────────────────────                        │
  │  Entry:  /admin/role-templates/{id}/apply                           │
  │  Actor:  Super Admin only                                           │
  │  Writes: module_role_permissions (role-scoped)                      │
  │  Safety: diff preview + confirmation + dangerous flag               │
  │  Audit:  event='template_applied', subject=Role                     │
  │                                                                     │
  │       │                                                             │
  │       │ (template configures role permissions)                      │
  │       ▼                                                             │
  │                                                                     │
  │  ROLE ASSIGNMENT (User Create / Edit)                               │
  │  ───────────────────────────────────────────                        │
  │  Entry:  /users/create  /users/{id}/edit                            │
  │  Actor:  Super Admin only                                           │
  │  Writes: user_role pivot table (user-scoped)                        │
  │  Reads:  module_role_permissions (for informational role preview)   │
  │  Safety: preventSuperAdminAssignment guard                          │
  │  Audit:  event='created'/'updated', subject=User                    │
  │                                                                     │
  │       │                                                             │
  │       │ (user assigned to role with pre-configured permissions)     │
  │       ▼                                                             │
  │                                                                     │
  │  USER OVERRIDES (User Create / Edit)                                │
  │  ───────────────────────────────────────────                        │
  │  Entry:  /users/create  /users/{id}/edit (collapsible)              │
  │  Actor:  Super Admin only                                           │
  │  Writes: user_module_permissions (user-scoped)                      │
  │  Reads:  module_role_permissions (for informational display)        │
  │  Safety: optional, collapsed by default, override wins              │
  │  Audit:  event='created'/'updated', properties include overrides    │
  │                                                                     │
  │       │                                                             │
  │       │ (per RBAC resolution: override → role permission → false)   │
  │       ▼                                                             │
  │                                                                     │
  │  ACCESS CONTROL (canOnModule)                                       │
  │  ───────────────────────────────────────────                        │
  │  Reads:  user_module_permissions → module_role_permissions          │
  │  Returns: boolean permission check for (user, module, action)       │
  │  Logic:  super-admin bypass → user override → role → false          │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

  KEY ARCHITECTURAL RULES:

  1. Role Templates ONLY write to module_role_permissions.
     They NEVER touch user_role, user_module_permissions, or users.

  2. User Create/Edit ONLY write to users, user_role, user_module_permissions.
     They NEVER touch module_role_permissions.

  3. Clone Existing User ONLY copies user_role and user_module_permissions.
     It NEVER touches module_role_permissions.

  4. The Role Preview in the create form is READ-ONLY.
     It shows what each selected role grants (from module_role_permissions).
     It does NOT modify anything.

  5. Audit events are scoped by resource type:
     - Role-scoped changes → subject = Role
     - User-scoped changes → subject = User

  6. No cross-layer writes. No hidden side effects.
     Each operation's blast radius is limited to its own layer.

SUMMARY
================================================================================

The architectural concern is VALID. The current plan's Step 2 (applying a Role
Template during User creation) would mutate ModuleRolePermission for the user's
selected role, affecting ALL users with that role — not just the new user.

The recommended architecture separates concerns strictly:

  - Role Templates → Role Configuration (writes to module_role_permissions)
  - User Creation → Role Assignment + Optional Clone + Optional Overrides
    (writes to users, user_role, user_module_permissions only)

This eliminates the risk of accidental role-wide permission changes during
user creation, simplifies the implementation (3-5 days vs 7-9), reduces test
count (10 vs 18), and produces a cleaner audit trail.

The "Quick Setup" section is replaced with an informational "Role Preview" that
shows what each selected role grants without modifying anything.
