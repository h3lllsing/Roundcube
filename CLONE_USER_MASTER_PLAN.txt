CLONE EXISTING USER — MASTER PLAN
==================================
Date: 2026-06-25
Depends on: RBAC v1.0 (frozen), user CRUD

============================================
BUSINESS PURPOSE
============================================

Allow a Super Admin to create a new user by copying roles and permission
overrides from an existing employee. Eliminates manual re-selection of
roles and 26-module × 8-permission matrix when onboarding a replacement or
additional team member with identical access requirements.

Target: new user created in under 30 seconds with 3 form fields + 4 checkboxes.

============================================
WHAT IS CLONED
============================================

Always cloned (no checkbox):
  - Roles (user_roles pivot rows)
  - User module permission overrides (user_module_permissions rows)

Checkbox-optional (default: checked):
  - Copy roles           [✓]
  - Copy overrides       [✓]
  - Copy status          [ ]  — copies suspended_at value
  - Copy department      [ ]  — copies department_id if department column exists

NEVER cloned:
  - Password                 — always required for new user (min 8 chars)
  - remember_token           — session-specific
  - email_verified_at        — new user must verify
  - Login sessions           — no session data copied
  - MFA / 2FA tokens         — not implemented; safeguard for future
  - created_at / updated_at  — auto-set by Eloquent
  - deleted_at               — new user is never soft-deleted
  - API tokens (Sanctum)     — each user generates their own
  - Activity log entries     — causer/subject references would be invalid
  - Owned records            — domains, tasks, notes, vault entries, etc.
                               remain owned by the original user
  - Notifications            — personal to the original user

============================================
UI FLOW
============================================

=== Entry Points ===

  1. Users index page — "Clone" icon button on each row
     Position: between View (eye) and Edit (pencil) icon buttons
     Icon: copy/document-duplicate SVG
     Title: "Clone user"

  2. User show page — "Clone User" button in the action header
     Position: next to Edit button
     Style: outline variant, secondary color

=== Clone Form ===

  Route: GET /users/{id}/clone  → UserController@cloneForm

  Layout: existing create.blade.php structure, modified.

  Header banner (amber/warning):
    "Cloning from: <source_user_name> (<source_user_email>)"

  Step/Form sections:

  1. BASIC INFO (always required):
     - Name *           [text input]
     - Email *          [email input, validated unique]
     - Password *       [password input, min 8]
     - Confirm Password * [password input, matches]

  2. CLONE OPTIONS (4 checkboxes):
     [✓] Copy roles — clones all user_role assignments
     [✓] Copy overrides — clones all user_module_permission rows
     [ ] Copy status — copies suspended_at value (suspended/active state)
     [ ] Copy department — copies department_id (only shown if department
         column exists on users table; hide otherwise)

  3. SUMMARY (read-only preview, updated via JS when checkboxes toggle):
     - Source user: <name>
     - Roles to copy: <role1>, <role2>, ... (N roles)
     - Overrides to copy: <module_name> (N modules with non-inherit values)
     - Status to copy: Active/Suspended
     - Department to copy: <name> or —

  4. SUBMIT:
     "Clone User" primary button
     "Cancel" secondary button → back to user show/index

  Client-side validation:
    - Name required
    - Email required, valid format
    - Password required, min 8 chars, matches confirmation
    - Warn if cloning a super-admin (modal confirmation: "This user has
      the super-admin role. Cloning it will grant super-admin privileges
      to the new user. Are you sure?")

=== Success Result ===

  On success:
    - Flash message: "User cloned from <source_name> successfully."
    - Redirect to: users.show for the new user
    - If clone failed (transaction rollback):
      - Flash error with details
      - Return to clone form with entered values preserved

=== Error States ===

  1. Source user not found → 404
  2. Source user is the only super-admin and cloning is attempted
     without confirmation → error: "Cannot clone the last super-admin
     without explicit confirmation."
  3. Email already taken → validation error on email field
  4. Password too weak → validation error

============================================
CONTROLLER METHODS
============================================

File: app/Http/Controllers/Web/UserController.php

Two new methods:

=== 1. cloneForm(int $id): View ===

  Route: GET /users/{id}/clone
  Middleware: super-admin (abort_unless)
  Logic:
    - Find source user or 404
    - Load source user's roles (for summary display)
    - Load source user's module permission override count (for summary)
    - Return clone form view with:
      - $sourceUser (the user being cloned)
      - $sourceRoles collection
      - $overrideCount (integer)
      - $modules (all modules with features, for optional override preview)

=== 2. cloneStore(Request $request, int $id): RedirectResponse ===

  Route: POST /users/{id}/clone
  Middleware: super-admin
  Logic:

    $sourceUser = User::findOrFail($id);
    $validated = $request->validate([
        'name'             => 'required|string|max:255',
        'email'            => 'required|email|max:255|unique:users,email',
        'password'         => 'required|string|min:8|confirmed',
        'copy_roles'       => 'boolean',
        'copy_overrides'   => 'boolean',
        'copy_status'      => 'boolean',
        'copy_department'  => 'boolean',
    ]);

    // Safety check: warn if cloning super-admin
    $hasSuperAdminRole = $sourceUser->hasRole('super-admin');
    if ($hasSuperAdminRole && !$request->boolean('confirm_super_admin')) {
        return back()->withErrors([
            'confirm_super_admin' => 'Confirm cloning super-admin role.'
        ]);
    }

    DB::transaction(function () use ($validated, $sourceUser, $request) {
        // 1. Create new user
        $newUser = User::create([
            'name'     => $validated['name'],
            'email'    => $validated['email'],
            'password' => Hash::make($validated['password']),
        ]);

        // 2. Clone roles
        if ($request->boolean('copy_roles')) {
            $roleIds = $sourceUser->roles()->pluck('roles.id')->toArray();
            $newUser->roles()->sync($roleIds);
        }

        // 3. Clone user module permission overrides
        if ($request->boolean('copy_overrides')) {
            $overrides = UserModulePermission::where('user_id', $sourceUser->id)->get();
            foreach ($overrides as $override) {
                UserModulePermission::create([
                    'user_id'     => $newUser->id,
                    'module_id'   => $override->module_id,
                    'can_create'  => $override->can_create,
                    'can_read'    => $override->can_read,
                    'can_update'  => $override->can_update,
                    'can_delete'  => $override->can_delete,
                    'can_approve' => $override->can_approve,
                    'can_export'  => $override->can_export,
                    'can_reveal'  => $override->can_reveal,
                    'can_import'  => $override->can_import,
                ]);
            }
        }

        // 4. Clone status (suspended_at)
        if ($request->boolean('copy_status')) {
            $newUser->suspended_at = $sourceUser->suspended_at;
            $newUser->save();
        }

        // 5. Clone department (if column exists)
        if ($request->boolean('copy_department') && $sourceUser->department_id) {
            $newUser->department_id = $sourceUser->department_id;
            $newUser->save();
        }

        // 6. Log the clone event
        activity()
            ->event('cloned')
            ->performedOn($sourceUser)
            ->causedBy(Auth::user())
            ->withProperties([
                'target_user_id'   => $newUser->id,
                'target_user_name' => $newUser->name,
                'target_user_email'=> $newUser->email,
                'copied_roles'     => $request->boolean('copy_roles'),
                'copied_overrides' => $request->boolean('copy_overrides'),
                'copied_status'    => $request->boolean('copy_status'),
                'copied_department'=> $request->boolean('copy_department'),
            ])
            ->log('User cloned: ' . $sourceUser->email . ' → ' . $newUser->email);

        // Store new user ID for redirect outside transaction
        $request->session()->flash('cloned_user_id', $newUser->id);
    });

    $newUserId = $request->session()->get('cloned_user_id');
    return redirect()
        ->route('users.show', $newUserId)
        ->with('success', 'User cloned from ' . $sourceUser->name . ' successfully.');

============================================
SAFETY RULES
============================================

Rule 1 — Super Admin cloning protection:
  Cloning a user with super-admin role requires explicit checkbox
  confirmation ("I confirm this user needs super-admin access").
  Without confirmation, the cloneStore method returns validation error.
  The role IS still cloned if confirmed — this is intentional (cloning
  a super-admin without their privileges defeats the purpose).

Rule 2 — Password never cloned:
  Enforced at the form level (required field) and at the data level
  (never read $sourceUser->password).

Rule 3 — Email uniqueness:
  Standard Laravel unique validation on users.email table.
  No fallback — the email must be unique.

Rule 4 — Transactional integrity:
  Entire clone operation wraps in DB::transaction(). If any step
  fails (e.g., role sync throws), all changes roll back. No partial
  clones.

Rule 5 — No side effects on source user:
  The clone operation is READ-ONLY on the source user. No touch(),
  no updated_at bump, no activity logged on the source user's
  activity timeline (the clone event is logged separately).

Rule 6 — Maximum one clone at a time:
  No batch clone. Each clone requires individual review of the
  clone options checkboxes. Batch user creation should use the
  Bulk Import feature (Phase 3, Feature 7).

Rule 7 — Audit trail:
  Every clone creates an activity log entry with event='cloned'
  on the source user record, recording target user info and which
  options were copied. This provides an audit trail for "where
  did this new user's permissions come from."

============================================
SUPER-ADMIN CLONING — DETAILED
============================================

Scenario: Source user has the super-admin role.

  Default behavior:
    - cloneForm shows warning banner (amber):
      "This user is a Super Admin. Cloning will grant full
      unrestricted access to the new user."
    - An additional checkbox appears below the Clone Options:
      [ ] I confirm — Clone super-admin role
    - Without checking this, cloneStore returns error:
      "Confirm cloning super-admin role."

  Best practice recommendation (documented in help text):
    "Instead of cloning a super-admin, consider cloning an Admin
      user and adjusting permissions as needed. Super-admin access
      should be granted sparingly."

  Edge case: Last super-admin cannot be cloned without confirmation.
    - The controller checks if the source user is the ONLY super-admin.
    - If yes, the warning is elevated to a blocking warning:
      "This is the only Super Admin. Cloning is allowed, but ensure
      you do not lose access to this account."

ROLLBACK PLAN
============================================

If the clone feature must be rolled back after deployment:

=== Rollback: Code ===
  1. Remove two controller methods (cloneForm, cloneStore)
  2. Remove two routes from web.php
  3. Remove clone button from users/index.blade.php
  4. Remove clone button from users/show.blade.php
  5. Delete the clone form view (users/clone.blade.php)

=== Rollback: Data ===
  1. Activity log entries with event='cloned' remain in activity_log
     table. They do not affect system behavior. Optional cleanup:
       DB::table('activity_log')->where('event', 'cloned')->delete();
  2. Users created via clone remain in the users table as normal users.
     They are valid users with their own roles and permissions.
     Deleting them:
       User::where('email', 'like', '%@cloned-%')->orWhere(...)
     is NOT recommended unless all cloned users are identified.
     Recommended: leave them — they are legitimate user accounts.

=== Rollback: Tests ===
  1. Delete tests/Feature/UserCloneTest.php
  2. Verify remaining RBAC tests still pass (167 tests)

=== Verification ===
  After rollback:
    1. php artisan test --filter RbacPhase  (all 167 must pass)
    2. Manual: verify users index page loads without clone button
    3. Manual: verify /users/{id}/clone returns 404
