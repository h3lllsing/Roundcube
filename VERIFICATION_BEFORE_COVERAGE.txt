4. SECURITY VERIFICATION
================================================================================

RISKS FOUND (ordered by severity):

=== HIGH (fix needed) ===

H1. API ImportController — NO super-admin authorization
    File:   app/Http/Controllers/Api/ImportController.php
    Risk:   Any authenticated user can POST /api/import/{type} to bulk-import
            CSV data into any of 17 model types including users, features,
            modules, webhooks, roles, and privileges. Route is only under
            auth:sanctum, NOT role:super-admin.
    Fix:    Add abort_unless(hasRole('super-admin'), 403) at the top of
            the store() method.

H2. VPS/Hosting/VoIP/ServiceProvider/DomainEmail/OtherService index queries
    select 'password' column
    Files:  Multiple Web controllers (VpsController, HostingController,
            ServiceProviderController, DomainEmailController,
            OtherServiceController, VoipController)
    Risk:   Plain-text passwords loaded into memory and potentially exposed
            in Blade views, browser cache, or server logs on index pages.
            These are NOT encrypted (unlike vault entries which use encrypted
            cast).
    Note:   This was introduced by the recent L13 select-columns optimization.
            The 'password' column was included in the select list for index
            views that display a "copy password" button. Consider whether
            the password is actually needed for the index page rendering or
            if it can be fetched via an async endpoint.
    Fix:    Remove 'password' from ->select() in index queries where the
            view uses a separate async fetch for password display.

=== MEDIUM (should fix) ===

M1. API ExportController — missing module-level permission check
    File:   app/Http/Controllers/Api/ExportController.php
    Risk:   Web ExportController checks canOnModule($module, 'export')
            for non-super-admin; API version does not.
    Fix:    Add the same canOnModule() check to the API export method.

M2. Non-Vault password reveals — no audit trail
    Files:  VpsController::getPassword, HostingController::getPassword,
            VoipController::getPassword/extensionPassword,
            ServiceProviderController::getPassword,
            DomainEmailController::getPassword,
            OtherServiceController::getPassword
    Risk:   These endpoints return plain-text passwords but produce NO
            activity log entry. Vault reveals are properly audited.
    Fix:    Add ->activity()->log() call to each password reveal method.

M3. Suspend/unsuspend — not audit-logged
    Files:  Web UserController::suspend/unsuspend (lines 173-191)
            API UsersController::suspend/unsuspend (lines 217-233)
    Risk:   Suspending/unsuspending a user is a high-impact operation
            with no audit trail.
    Fix:    Add activity log entry after suspend/unsuspend.

=== LOW (nice to fix) ===

L1. Web DashboardController — totalNotes leaks system-wide count
    File:   app/Http/Controllers/Web/DashboardController.php (line 84)
    Risk:   Any authenticated user sees total note count for all users.
    Fix:    Remove or filter by user_id.

L2. API DashboardController — total_notes leaks system-wide count
    File:   app/Http/Controllers/Api/DashboardController.php (line 96)
    Risk:   Same as L1.
    Fix:    Remove or filter by user_id.

L3. Web MonitorController — no rate limiting
    File:   app/Http/Controllers/Web/MonitorController.php
    Risk:   No throttle on monitor check endpoint.
    Fix:    Add ->middleware('throttle:10,1') to monitor web route.

L4. Web Feature/Module index — no permission filtering
    Files:  Web FeatureController::index, ModuleController::index
    Risk:   Any user can list all features/modules.
    Note:   API versions do filter by permissions. Low severity since
            features/modules are not sensitive.

L5. Web Role/Privilege controllers — no inline auth (defense-in-depth)
    Files:  Web RoleController, PrivilegeController
    Fix:    Consider adding abort_unless(hasRole('super-admin'), 403) as
            defense-in-depth. Currently only route-protected.
