ROLE TEMPLATES — MASTER PLAN
============================
Date: 2026-06-25
Depends on: RBAC v1.0 (frozen)

This document defines the Role Templates feature. Templates are pre-configured
permission blueprints that a Super Admin can apply to any role in one click.
They do NOT replace ModuleRolePermission — they only pre-fill it.

============================================
ARCHITECTURE RULE
============================================

Templates do NOT create a new permission layer. Applying a template copies
permission values into the existing ModuleRolePermission table. The RBAC
resolution logic (canOnModule, getEffectiveModulePermissions, etc.) is
completely unchanged. All existing RBAC v1.0 rules continue to apply.

============================================
DB CHANGES REQUIRED
============================================

=== Table: role_templates ===

  id              bigIncrements
  name            string(255)
  slug            string(255), unique
  description     text, nullable
  is_protected    boolean, default false
                    true = cannot be edited or deleted via UI
  created_at      timestamp
  updated_at      timestamp

=== Table: role_template_modules ===

  id              bigIncrements
  role_template_id  FK -> role_templates (cascade)
  module_id         FK -> modules (cascade)
  can_create      boolean, not null, default false
  can_read        boolean, not null, default false
  can_update      boolean, not null, default false
  can_delete      boolean, not null, default false
  can_approve     boolean, not null, default false
  can_export      boolean, not null, default false
  can_reveal      boolean, not null, default false
  can_import      boolean, not null, default false
  created_at      timestamp
  updated_at      timestamp

  UNIQUE: role_template_id, module_id

No changes to existing RBAC tables (module_role_permissions,
user_module_permissions, users, roles).

============================================
APPLY TEMPLATE TO ROLE — LOGIC
============================================

When a Super Admin applies a template to a role:

  1. Fetch all rows from role_template_modules for the template.
  2. For each module row:
     a. UPSERT into module_role_permissions:
        role_id = target role's id
        module_id = template module's id
        All 8 permission columns = template values
  3. If the role already has permission rows for some modules,
     they are OVERWRITTEN (upsert by unique key module_id + role_id).
  4. Existing permission rows for modules NOT in the template
     are UNCHANGED (template only touches its defined modules).
  5. No changes to any user_module_permissions rows (user overrides
     are preserved and continue to take precedence per RBAC rules).

Implementation: single DB transaction, batch upsert.

Option — "Strict Apply" (future enhancement):
  If enabled, after step 4, DELETE any module_role_permissions rows
  for the role where the template has no entry. This ensures the
  role's permissions exactly match the template. Default: off.

============================================
RISKS
============================================

1. Template Drift
   New modules added after template creation are not included. When
   applied, the template won't touch those modules, leaving whatever
   permissions the role already has (or none).
   Mitigation: "Sync to current modules" button on template edit that
   adds missing modules with default-denied permissions.

2. Overwriting Without Recovery
   Apply overwrites existing ModuleRolePermission rows. There is no
   undo. Mitigation: show diff preview before apply; optionally add
   a "backup" feature that saves current role permissions to a JSON
   log before overwriting.

3. Protected Template Bypass
   is_protected is a DB flag, not an absolute guarantee. Anyone with
   direct DB access can change it. Mitigation: enforce in the controller
   (abort 403 for edit/delete on protected) and in validation.

4. Permissions Not Reflecting Reality
   A template grants maximum possible permissions, but user overrides
   can restrict individual users. Super Admin must understand that
   applying a template to a role is not the same as applying it to
   every user with that role.
