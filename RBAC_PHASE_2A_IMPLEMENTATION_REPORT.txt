1.  MIGRATION CREATED
    -------------------------------------------------------------------------
    File: database/migrations/2026_06_25_000005_create_user_module_permissions_table.php

    Table: user_module_permissions

    Columns:
      id              bigint(20) unsigned  PK, auto_increment
      user_id         bigint(20) unsigned  FK → users, CASCADE ON DELETE
      module_id       bigint(20) unsigned  FK → modules, CASCADE ON DELETE
      can_create      tinyint(1)           NULLABLE (default NULL = inherit)
      can_read        tinyint(1)           NULLABLE
      can_update      tinyint(1)           NULLABLE
      can_delete      tinyint(1)           NULLABLE
      can_approve     tinyint(1)           NULLABLE
      can_export      tinyint(1)           NULLABLE
      can_reveal      tinyint(1)           NULLABLE  ← NEW (password reveal)
      can_import      tinyint(1)           NULLABLE  ← NEW (data import)
      created_at      timestamp
      updated_at      timestamp

      UNIQUE (user_id, module_id)

    Key design:
      - NULL  → inherit from role permission (fall through to module_role_permissions)
      - true  → explicit allow (overrides role, even if role denies)
      - false → explicit deny (overrides role, even if role grants)

4.  TRAIT METHODS CHANGED
    -------------------------------------------------------------------------
    File: app/Traits/HasModulePermissions.php

    All four core permission methods were updated with merge logic:

    (a) canOnModule(Module $module, string $action): bool
        Before: Query module_role_permissions by role_ids only
        After:  Check user_module_permissions first;
                if column is NOT NULL → return its value (true=allow, false=deny);
                if NULL or no row → fall through to role-based module_role_permissions

    (b) getModulePermissions(Module $module): ?array
        Before: Return merged role permissions for the 6 original columns
        After:  Return role permissions, then overlay user overrides for any
                column where user override is non-null

    (c) getAllModulePermissions(): array
        Before: Group module_role_permissions by module_id, merge across roles
        After:  Same merge, then apply user overrides on top; includes modules
                that only have user overrides (no role permission row)

    (d) getAccessibleModuleIds(string $action): array
        Before: Module IDs where ANY role has can_{action}=true
        After:  Start with role-based IDs;
                ADD module IDs where user override grants (true);
                REMOVE module IDs where user override denies (false);
                Return unique sorted array

    (e) NEW: getUserModuleOverride(Module $module): ?array
        Returns the raw user override row as an array of 8 nullable booleans,
        or null if no override row exists.

    (f) NEW: getEffectiveModulePermissions(Module $module): array
        Returns detailed per-permission breakdown:
          [
            'can_read' => [
              'role'          => true|false|null,   // from module_role_permissions
              'user_override' => true|false|null,   // from user_module_permissions
              'effective'     => true|false,        // final resolved value
              'source'        => 'Role' | 'User Override' | 'None',
            ],
            ...
          ]

    (g) canAccessVault():  Unchanged (still uses role-based can_read only)
    (h) isVaultOwner():    Unchanged

5.  USER CONTROLLER UPDATED
    -------------------------------------------------------------------------
    File: app/Http/Controllers/Web/UserController.php

    Changes:

    (a) New private helper: getModulesWithFeatures()
        Returns all modules with feature relation, ordered by name.

    (b) New private helper: saveUserModulePermissions(User $user, ?array $permissions)
        - If permissions array is null → no-op
        - For each (moduleId → [permission => value]):
          - '1' / 1 / true  → set true
          - '0' / 0 / false → set false
          - '' / null       → set null (inherit)
        - If ANY column has a non-null value → updateOrCreate the row
        - If ALL columns are null/inherit → DELETE the override row

    (c) New private helper: preventSuperAdminAssignment(Request $request)
        Aborts with 403 if request contains super-admin role ID.

    (d) create() action:
        Now also passes $modules and $userOverrides (empty) to the view.

    (e) store() action:
        Now validates 'permissions' as nullable array.
        Calls preventSuperAdminAssignment().
        Calls saveUserModulePermissions() inside the transaction.

    (f) show() action:
        Completely rewritten:
        - Uses getEffectiveModulePermissions() for detailed source tracking
        - Passes $modulePermissions with 'source' info for each permission
        - Shows effective value + "override" label when user override active

    (g) edit() action:
        Now also passes $modules and $userOverrides (existing overrides keyed
        by module_id) to the view.

    (h) update() action:
        Now validates 'permissions' as nullable array.
        Calls preventSuperAdminAssignment().
        Includes self-demotion protection:
          "Cannot remove your own Super Admin role."
        Calls saveUserModulePermissions() inside the transaction.

    (i) destroy() action:
        Added last-super-admin protection:
          "Cannot delete the last Super Admin user."
          Checks if target user has super-admin role, counts total super-admins.
          Only blocks if count ≤ 1.

6.  USER CREATE/EDIT UI CHANGED
    -------------------------------------------------------------------------
    Files:
      resources/views/users/create.blade.php
      resources/views/users/edit.blade.php
      resources/views/users/show.blade.php

    Create/Edit views now include a "Special Per-User Permissions" section
    below the roles section.  It renders a table:

      Module | Feature | C | R | U | D | A | E | Reveal | Import

    Each cell is a 3-state <select> dropdown:
      — (Inherit)  → stored as NULL → role permission applies
      ✓ (Allow)    → stored as true → explicit grant
      ✗ (Deny)     → stored as false → explicit deny

    Show view now renders "Effective Module Permissions" with the same 8
    columns plus a source indicator.  Permissions overridden by user-level
    settings show an "override" badge.
