1. EXECUTIVE SUMMARY
================================================================================

This is a Laravel 12 + Sanctum application with a custom module-level RBAC layer
built on top of the HasinHayder/Tyro role/privilege package. It manages 10+
service entity types (Domain, Hosting, VPS, VoIP, ServiceProvider, Vault, etc.)
with full CRUD + bulk operations + export/import + monitoring + notifications.

The application is structurally sound with clear MVC separation, good test
coverage (1700 tests), and strong security fundamentals (encrypted secrets,
RBAC, audit trails, rate limiting). It is approaching production readiness
but has several critical issues that must be resolved before deployment.

PROJECT MATURITY SCORE: 7.2 / 10
  - Functional completeness:  9/10 (near-feature-complete)
  - Code quality (SOLID):     6/10 (SRP/DRY violations, no interfaces)
  - Security posture:         7/10 (critical gaps: no email verification,
                                   suspended user enforcement missing)
  - Test coverage:            8/10 (1700 tests, 4294 assertions, good)
  - Performance:              6/10 (N+1 queries, no queuing, no chunking)
  - Scalability:              5/10 (file sessions, sync sends, no caching)
  - Documentation:            4/10 (no API docs beyond Swagger schemas,
                                   no architecture docs, no SECURITY.md)

3. PRODUCTION READINESS ASSESSMENT
================================================================================

  OVERALL: NOT READY

  CRITICAL (Must fix before production):

  [C1] laravel/tinker in production require -- composer.json
       Tinker is an interactive PHP REPL. If accidentally exposed
       (e.g., via composer install --no-dev not being run), an
       attacker can execute arbitrary PHP code.

  [C2] APP_DEBUG=true & APP_ENV=local in .env
       Exposes full stack traces and environment dumps to end users.

  [C3] Hardcoded secrets in .env (DB password, APP_KEY)
       The .env file is visible in the working directory. Secrets
       are not rotated and are committed to version control.

  HIGH (Strongly recommended before production):

  [H1] No HTTPS enforcement on session cookies
       SESSION_SECURE_COOKIE not set. Sessions can be transmitted
       over HTTP.

  [H2] Session encryption disabled (SESSION_ENCRYPT=false)
       Session data stored in plaintext in file-based sessions.

  [H3] FTP deployment with SSL disabled (deploy.sh)
       Credentials and files transferred in plaintext.

  [H4] No maintenance mode during deployment
       Users see broken state during deploy.

  [H5] No CI/CD pipeline
       No automated testing or deployment pipeline.

  [H6] No email verification
       MustVerifyEmail is commented out. Users register and
       immediately access the app without verifying ownership
       of the email address.

  [H7] Suspended user enforcement not implemented
       isSuspended() exists on User model but is never checked
       in middleware or controllers. Suspended users with active
       tokens can still access all resources.

  MEDIUM:

  [M1] MAIL_MAILER=log -- emails silently discarded
  [M2] File-based cache (CACHE_STORE=file) -- not shared across servers
  [M3] No SECURITY.md or vulnerability disclosure policy
  [M4] robots.txt allows crawling of /api/, /docs/ paths
  [M5] Docker volume mount of entire source tree (anti-pattern)
  [M6] Stale .env.example.bak file

12. ARCHITECTURE SMELLS & FUTURE RISKS
================================================================================

  SMELL 1: Dual notification systems
  ────────────────────────────────────────────────────────────────────────
  ExpiryNotificationService (in-app DB notifications, expiry:check
  command) and RenewalNotificationService (email reminders,
  renewals:send-email-reminders) both target ExpiryTracker with similar
  trigger logic but completely separate code paths. Users may receive
  overlapping notifications with different timing.

  SMELL 2: Tight coupling to ExpiryTracker
  ────────────────────────────────────────────────────────────────────────
  RenewalNotificationService is exclusively bound to ExpiryTracker model.
  Every public method takes ExpiryTracker as parameter. No interface, no
  generic "Expirable" contract. Adding Domain/Hosting/VPS email reminders
  requires rewriting the service.

  SMELL 3: Global config() mutation in queue context
  ────────────────────────────────────────────────────────────────────────
  resolveMailer() mutates config('mail.mailers.smtp_profile') at runtime.
  If two queue jobs run in the same worker process, their profile configs
  will collide. Currently only used in CLI (safe), but future queue usage
  will break.

  SMELL 4: N+1 quagmire
  ────────────────────────────────────────────────────────────────────────
  4+ locations with N+1 query patterns:
   - SidebarComposer (12 queries per page load)
   - canOnModule() (2 queries per call)
   - SmtpProfile index (usageCount in loop)
   - getAccessibleModuleIds() (3-4 queries per call)
  At scale (100s of concurrent users), these will cause DB load spikes.

  SMELL 5: No application-level caching for permissions
  ────────────────────────────────────────────────────────────────────────
  ModuleRolePermission queries hit the database on every request. No
  cache layer despite permissions changing infrequently.

  SMELL 6: Synchronous email sending at scale
  ────────────────────────────────────────────────────────────────────────
  No queue, no chunking, no batching. The 02:00 cron will become a
  bottleneck as tracker count grows. 10,000 trackers x 3 recipients =
  30,000 sequential SMTP connections.

  SMELL 7: Inconsistent permission check patterns
  ────────────────────────────────────────────────────────────────────────
  4 different patterns used (route middleware, inline hasRole, inline
  canOnModule, @hasrole in Blade). Some controllers check super-admin
  before module perms; some check module perms first. The API
  ExportController skips can_export for admin roles entirely.

  SMELL 8: Controllers doing business logic
  ────────────────────────────────────────────────────────────────────────
  UserController (511 lines), DashboardController::computeDashboardData()
  (136 lines), RoleTemplateController::executeApply() (71 lines) all
  handle business logic that belongs in service classes.

  FUTURE RISKS:
  - Adding a 3rd notification system for other entity types (Domain,
    Hosting) would exacerbate the dual-notification confusion
  - Permission caching gap will become critical at 1000+ concurrent users
  - File-based sessions+cache prevent horizontal scaling
  - No CI/CD pipeline means no automated regression checks on deploy
  - FTP-based deployment with SSL disabled is a security incident waiting
    to happen
