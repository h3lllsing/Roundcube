2. PERMISSION MODEL
==========================================

Three-tier architecture:
  A. Super Admin — bypasses ALL permission checks
     Role slug: 'super-admin'
     Tests: hasRole('super-admin') → grants all access

  B. Role Permissions — baseline per-module settings for a role
     Table: module_role_permissions
     Columns: module_id, role_id + 8 boolean permission columns

  C. User Overrides — per-user exceptions on specific modules
     Table: user_module_permissions
     Columns: user_id, module_id + 8 nullable boolean permission columns
     Override logic: if column is not null, its value wins (true or false)
     If column is null, inherits from role permission

Permission resolution (canOnModule method in HasModulePermissions trait):
  1. Check UserModulePermission for user override (if exists and non-null → return)
  2. Check ModuleRolePermission for any role where column = true → return true
  3. Default: false

Super admin check happens BEFORE calling canOnModule in all guard code:
  if ($user->hasRole('super-admin')) { return; } // or return true

==========================================
3. ROLE PERMISSIONS (module_role_permissions)
==========================================

Table: module_role_permissions
Schema:
  id              bigIncrements
  module_id       FK -> modules (cascade)
  role_id         FK -> roles (cascade)
  can_create      boolean, default false
  can_read        boolean, default false
  can_update      boolean, default false
  can_delete      boolean, default false
  can_approve     boolean, default false
  can_export      boolean, default false
  can_reveal      boolean, default false
  can_import      boolean, default false
  created_at      timestamp
  updated_at      timestamp
  UNIQUE: module_id, role_id

One row per (module, role) pair. Multiple roles per user: any role with
true = user has that permission (OR logic across roles). The only
exception is getAccessibleModuleIds which merges overrides additively.

==========================================
4. USER OVERRIDES (user_module_permissions)
==========================================

Table: user_module_permissions
Schema:
  id              bigIncrements
  user_id         FK -> users (cascade)
  module_id       FK -> modules (cascade)
  can_create      boolean, nullable
  can_read        boolean, nullable
  can_update      boolean, nullable
  can_delete      boolean, nullable
  can_approve     boolean, nullable
  can_export      boolean, nullable
  can_reveal      boolean, nullable
  can_import      boolean, nullable
  created_at      timestamp
  updated_at      timestamp
  UNIQUE: user_id, module_id

Override resolution:
  null  → inherit from role permission
  true  → grant, overriding any role denial
  false → deny, overriding any role grant

Override management UI: User create/edit pages show Allow/Deny/Inherit
selects for each permission on each role-accessible module.

==========================================
5. CRUD ENFORCEMENT
==========================================

All 9 web resource controllers implement server-side guards:

  create/store:      abort_unless super-admin OR canOnModule($module, 'create')
  edit/update:       abort_unless super-admin OR canOnModule($module, 'update')
  destroy:           abort_unless super-admin OR canOnModule($module, 'delete')
  restore/forceDelete: super-admin only

Controllers: DomainController, HostingController, VpsController,
VoipController, ServiceProviderController, DomainEmailController,
OtherServiceController, VaultEntryController, ExpiryTrackerController

TaskController: ownership-scoped (created_by or assignee), no can_create check

UI visibility matches server-side guards:
  Index: Create button hidden when !can_create
  Index: Export button hidden when !can_export
  Index/Show: Edit button hidden when !can_update
  Index/Show: Delete button hidden when !can_delete
  Bulk actions hidden when no permitted actions

==========================================
6. EXPORT ENFORCEMENT
==========================================

ExportController:
  Super-admin: exports all records of any type
  Admin with can_export on a module: exports module-scoped records
  Normal user: exports records where user_id = own ID

Export types restricted:
  super-admin: all types (including 'users', 'roles', 'features', 'modules')
  Others: only module-based types where can_export is granted

9. PERMISSION INSPECTOR
==========================================

Route: GET /users/{id} (UserController@show, super-admin only)
Displays:
  User summary header: ID, Email, Roles, Status, Last Login
  6 summary stat cards: Roles, Accessible Modules, Denied Modules,
    Overrides, Allowed Permissions, Denied Permissions

  Permission Matrix table with ALL modules (including those without
    any role permissions)

    (purple), None (gray), Super Admin (green)

Backend: getEffectiveModulePermissions() in HasModulePermissions trait
  returns array with role, user_override, effective, source per permission.

11. FUTURE MODULE CHECKLIST
==========================================

When adding a new module entity to the system, the following steps are
required to fully integrate with RBAC v1.0:

  1. Add module entry to FeatureModuleSeeder (name + slug)
  2. Run seeder: php artisan db:seed --class=FeatureModuleSeeder
  3. ModuleRolePermission rows automatically created? NO — must be
     set up via RBAC UI (role edit → module permissions matrix).
  4. Data scoping: add module_id FK to the new model's migration
  5. Module model: add belongsTo(Module::class) relationship
  6. Module model: implement 3-tier scoping (see existing models for pattern)
  7. Controller: add server-side guards (can_create/update/delete) in

  8. Controller: add reveal guard if password field exists — see Phase 2B-3
  9. Index view: add @if($canCreate) around Create button — Phase 2C-1
  10. Index view: add @if($canExport) around Export button — Phase 2C-1
  11. Index/Show view: add Edit/Delete button guards — Phase 2C-2
  12. Show view: add password reveal/copy button guards — Phase 2C-3
      (if applicable)

      AND add flag in DashboardController computeDashboardData() — Phase 2C-5

12. ARCHITECTURE FREEZE RULE
==========================================

NO RBAC SCHEMA CHANGE WITHOUT ARCHITECTURE REVIEW.

The following are FROZEN and MUST NOT be modified without written
architecture review and approval:

  A. Table structures:
     - module_role_permissions (columns, types, constraints)
     - user_module_permissions (columns, types, constraints)
     - Any migration that adds/removes/renames permission columns

  B. Permission resolution logic:
     - canOnModule() method signature and behavior
     - getEffectiveModulePermissions() merge logic
     - getAccessibleModuleIds() merge logic
     - The override precedence: user override → role permission → false
     - Super-admin bypass: the rule that hasRole('super-admin') skips all checks

  C. Server-side guard pattern:
     - abort_unless super-admin OR canOnModule($record->module, 'action')
     - The 3-tier data scoping pattern (super-admin → admin → owner)
     - The rule that super-admin bypass is checked BEFORE canOnModule

The following MAY be changed without review:
  - UI presentation of permission data (colors, layout, badges)
  - Adding new filter/sort options to existing views
  - Adding new test cases
  - Adding new modules following the Future Module Checklist
  - Adding documentation, reports, or read-only views of permission data

Reason: RBAC v1.0 has 167 passing tests covering all enforcement layers.
Any schema or logic change risks cascading failures across all phases.
Changes require full regression test run (all 167 tests must pass).
