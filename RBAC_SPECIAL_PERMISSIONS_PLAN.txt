1. CURRENT SYSTEM ARCHITECTURE (Verified)
   -------------------------------------------------------------------------
   Two-layer RBAC:

     LAYER 1 — Tyro (vendor/hasinHayder/tyro)
       Tables: roles, user_roles (pivot), privileges, privilege_role (pivot)
       Methods: $user->hasRole('super-admin'), $user->assignRole($role),
                $user->hasPrivilege('users.manage')
       One user CAN have multiple roles (BelongsToMany).
       Users are assigned roles; roles have privileges (string-based).

     LAYER 2 — Custom App Module Permissions
       Table: module_role_permissions
         Columns: module_id, role_id, can_create, can_read, can_update,
                  can_delete, can_approve, can_export
         Unique: (module_id, role_id)
       Trait: app/Traits/HasModulePermissions.php
         Methods: getAccessibleModuleIds($action), canOnModule($module, $action),
                  getModulePermissions($module), getAllModulePermissions(),
                  canAccessVault($vault), isVaultOwner($vault)
       This is a role→module permission matrix.  Permissions are MERGED
       across all roles a user has (OR logic — if ANY role grants it, it's true).

   Scoping (Phase 1 complete):
     Web controllers use RbacScope::apply() which calls
     getAccessibleModuleIds('read') for admin users to add a global
     whereIn('module_id', ...) scope.

   Password protection:
     VaultService encrypts passwords (Crypt::encryptString).
     Non-vault passwords stored in plaintext columns.
     All reveals audit-logged with event 'revealed'.
     Vault reveals also dispatch notification + fire webhook.
     Throttled at 10/min.

2. CAN EXISTING ROLE + MODULE_ROLE_PERMISSIONS SUPPORT THE 5 ROLES?
   -------------------------------------------------------------------------
   YES — partially.  Here is the mapping:

     (a) Super Admin
         Already implemented.
         Route middleware: role:super-admin.
         Inline: abort_unless(hasRole('super-admin'), 403).
         No module_role_permissions needed (bypasses all scoping).

     (b) Admin
         Already supported.
         Role slug: admin.
         Module permissions set via module_role_permissions.
         RbacScope::apply() handles scoping at query level.
         SECURITY GAP: Admin currently has full CRUD via create/store/update/destroy
         methods because userOwnedFilter() only scopes read queries.  The
         can_create/can_update/can_delete flags are NOT enforced at the
         controller level (no abort_unless checks).

     (c) IT Support (new)
         Can be assigned the existing "user" role or a new custom role.
         Module_role_permissions can be set per-module.
         Default: can_read only, can_update optional per Super Admin.
         Password reveal: no dedicated permission exists yet.
         GAP: No user-level override mechanism.

     (d) Read Only (new)
         Can use existing "user" role or a new "read-only" role.
         Module_role_permissions: can_read=true, everything else=false.
         But restriction to specific modules (Software, Hosting, Domains,
         Domain Emails) would need selective module_role_permissions.
         GAP: No password_reveal permission column.

     (e) Special Per-User Permissions (new)
         NOT SUPPORTED by the current schema.
         module_role_permissions is keyed on (module_id, role_id) only.
         There is no user_id column.  Two users with the same role get
         identical permissions.  No override mechanism exists.

3. IS PER-USER SPECIAL PERMISSION POSSIBLE CURRENTLY?
   -------------------------------------------------------------------------
   NO.  There is no table, column, or mechanism to store user-specific
   permission overrides.

   The existing merger logic in HasModulePermissions trait:
     - canOnModule():     queries module_role_permissions WHERE role_id IN user.roles
     - getModulePermissions(): same — role-scoped only
     - getAllModulePermissions(): same — role-scoped only

   A user with two roles (e.g., admin + read-only) gets an OR-merge:
   if either role grants can_update, the user can update.  This is the
   only way to achieve differential permissions per user, but it requires
   creating N roles (one per permission combination), which does not scale.

4. OPTIONS ANALYSIS — SAFEST DESIGN FOR PER-USER PERMISSIONS
   -------------------------------------------------------------------------
   Sorted by safety, simplicity, and maintainability:

   ═══════════════════════════════════════════════════════════════════════
   OPTION A: user_module_permissions table (RECOMMENDED)
   ═══════════════════════════════════════════════════════════════════════

   New table: user_module_permissions
     user_id       FK → users (cascade delete)
     module_id     FK → modules (cascade delete)
     can_create    bool, default NULL
     can_read      bool, default NULL
     can_update    bool, default NULL
     can_delete    bool, default NULL
     can_approve   bool, default NULL
     can_export    bool, default NULL
     can_reveal    bool, default NULL    ← NEW column for password reveal
     can_import    bool, default NULL    ← NEW column for import
     timestamps
     UNIQUE (user_id, module_id)

   Logic change in HasModulePermissions trait:
     For each permission check, merge:
       role_permission (module_role_permissions) OR
       user_permission (user_module_permissions, if not null)
     The user-level permission acts as an OVERRIDE:
       - NULL  → fall through to role permission
       - true  → grant regardless of role
       - false → deny regardless of role

   Example pseudo-code for canOnModule():
     public function canOnModule(Module $module, string $action): bool
     {
         $column = 'can_'.$action;

         // check user-level override first
         $userPerm = UserModulePermission::where('user_id', $this->id)
             ->where('module_id', $module->id)
             ->first();
         if ($userPerm && $userPerm->$column !== null) {
             return $userPerm->$column;
         }

         // fall back to role-level
         return ModuleRolePermission::whereIn('role_id', $this->getRoleIds())
             ->where('module_id', $module->id)
             ->where($column, true)
             ->exists();
     }

   RISK: LOW — additive change, non-breaking.
   BENEFIT: Clean, queryable, auditable, easy UI.

   ═══════════════════════════════════════════════════════════════════════
   OPTION B: JSON permissions column on users table
   ═══════════════════════════════════════════════════════════════════════

   Add column: users.special_permissions JSON, nullable.
     e.g. {"module_overrides": {"1": {"can_update": true, "can_reveal": false}}}

   RISK: MEDIUM — JSON querying is messy, harder to audit, no FK constraints.
   BENEFIT: No new table.

   ═══════════════════════════════════════════════════════════════════════
   OPTION C: Use existing role system only
   ═══════════════════════════════════════════════════════════════════════

   Create many fine-grained roles (e.g., "it-support-hosting-edit",
   "it-support-domain-readonly").  Users get multiple roles.

   RISK: HIGH — role explosion, complex management UI, confusion.
   No per-user override — only per-role combinations.
   Not suitable for "one IT Support can edit, another cannot" scenario
   unless each user gets a unique role.

   ═══════════════════════════════════════════════════════════════════════

   RECOMMENDATION: Option A.  Add user_module_permissions table and merge
   logic in HasModulePermissions trait.  This is the least risky, most
   maintainable, and most auditable approach.

5. HOW TO KEEP SUPER ADMIN PROTECTED
   -------------------------------------------------------------------------
   Already in place:
     ✅ Route middleware: role:super-admin on all /users/*, /module-permissions/*,
        /features/*, /modules/*, /import/*, /activity-logs/*, /reports/*,
        /login-audits/* routes.
     ✅ Inline: abort_unless(hasRole('super-admin'), 403) in UserController
        (every method), ModulePermissionController, FeatureController, etc.
     ✅ Role dropdown in user create/edit views excludes 'super-admin' and '*'
        (wildcard) slugs.
     ✅ RbacScope::apply() bypasses all scoping for super-admin.

   Still needed:
     ☐ Prevent self-demotion: A super-admin should not be able to remove
        their own super-admin role in user edit form.
     ☐ Prevent last-super-admin deletion: At least one super-admin must
        always exist (enforce in UserController::destroy and update).
     ☐ Add audit logging for all user role/permission changes (currently
        no activity log when roles are changed).

6. HOW TO PREVENT ADMIN FROM EDITING SUPER ADMIN
   -------------------------------------------------------------------------
   Already done:
     ✅ User management routes (users.*) are entirely under the
        role:super-admin middleware group (web.php line 203).
        An Admin user cannot even reach /users/*.
     ✅ API UsersController has abort_unless(hasRole('super-admin'), 403).
     ✅ IMPORTANT: The create/edit views exclude super-admin role from the
        dropdown, so even a super-admin cannot accidentally assign the
        super-admin role to someone through the UI.

   No additional protection needed — the route-level guard is sufficient.

   However, for defense-in-depth:
     ☐ Add a check in UserController::update and store to reject
        role assignments containing super-admin role:
        if (in_array($superAdminRoleId, $request->roles ?? [])) abort(403);

9. TESTS NEEDED
   -------------------------------------------------------------------------
   Phase 2 tests:

     (a) UserModulePermission model tests
         - Can create with valid data
         - Unique (user_id, module_id) enforced
         - Nullable permission columns work correctly
         - Cascade delete when user/module is deleted

     (b) HasModulePermissions trait — merge logic
         - User with role permission only → inherits role permission
         - User with user_module_permissions set to true → overrides role
         - User with user_module_permissions set to false → denies despite role
         - User with null user_module_permissions → falls through to role
         - User with no role and no user permission → returns false

     (c) Controller enforcement
         - Web store/create → abort_unless(hasRole('super-admin') OR can_create)
         - Web update → abort_unless(can_update on the record's module)
         - Web destroy → abort_unless(can_delete on the record's module)
         - Web show → abort_unless(can_read on the record's module)
         - Password reveal → abort_unless(can_reveal on the record's module)

     (d) Super Admin protection
         - Cannot self-demote (remove own super-admin role)
         - Cannot delete last super-admin
         - Non-super-admin cannot access user management

     (e) Export fix
         - Admin with can_export sees all module-scoped records, not just own
         - Normal user with can_export sees only own records

     (f) Import permission
         - Admin with can_import can import into permitted module
         - Admin without can_import cannot import
