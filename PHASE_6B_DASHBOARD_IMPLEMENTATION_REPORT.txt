4. CACHE IMPLEMENTATION
-----------------------

  Cache driver: config-dependent (array in test, file/database/redis in prod)
  Bust mechanism: Cache::get('dashboard:version', 0) — increment to invalidate
                  all widget caches for all users
  Key format: dashboard:w:{WidgetClass::SLUG}:{user_id}:v{version}
  Per-widget TTLs:
    ActivityWidget       60s  (freshness sensitive)
    TasksWidget         120s  (moderate freshness)
    OperationsWidget    300s  (aggregate data)
    RenewalsWidget      300s
    AssetsWidget        300s
    VaultWidget         300s
    QuickActionsWidget  300s
    SmtpWidget          600s  (infrequently changed)
    ServerHealthWidget  600s

  Scheduler last-run: Cache::put('scheduler:last_run', now(), 86400)
                      stored in SendEmailReminders command

12. EXECUTIVE SUMMARY
---------------------

  The monolithic DashboardController (42+ queries per page load) was replaced
  with a widget-per-widget architecture. 9 flat widget classes in app/Dashboard/
  each own their data-fetching logic, cache TTL, and SLUG constant. The
  controller loops through $widgetClasses with per-widget Cache::remember,
  reducing warm-cache queries to 0-2 and cold-cache queries to ~16-18.

  All widgets include:
    - Empty-state messages in Blade partials
    - Role-appropriate data scoping
    - Individual cache TTLs
    - Graceful failure (widget exception logs warning, page continues rendering)

  Three MVP charts (Tasks doughnut, Assets doughnut, Renewals expiry bar) were
  added alongside the preserved Services by Type bar chart — all embedded
  inside their respective widget Blades with no standalone ChartsWidget.

  The scheduler last-run is tracked via Cache::put in SendEmailReminders with
  no new database table or migration.
