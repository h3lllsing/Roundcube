PHASE 3A-3: BETTER CREATE USER EXPERIENCE — PLAN
==================================================
Date: 2026-06-25
Depends on: RBAC v1.0 (frozen), Clone User (Phase 3A-1), Built-in Role Templates (Phase 3A-2)

This document defines the redesign of the User Create form for Super Admin, integrating
basic info, role templates, optional cloning, and special overrides into a single page.

============================================
DESIGN PRINCIPLES
============================================

1. Single page, not a wizard. All sections visible at once with appropriate defaults.
2. Sections are progressively disclosed: basic info always shown, quick setup next,
   clone section hidden by default, overrides collapsed by default.
3. Reuse existing logic — do not duplicate or modify Role Template apply or Clone User core.
4. Conflict resolution is deterministic: Clone → Template → Overrides (last wins).
5. User overrides always win over role/template values per RBAC v1.0 freeze rule.

============================================
CONFLICT RESOLUTION (ORDER OF OPERATIONS)
============================================

When multiple setup options are selected simultaneously, operations execute in this order:

  Step 1: CLONE
    If clone source is selected, roles and overrides are copied from source user
    to the new user. Password is NEVER copied.
    Result: new user has roles + optional overrides from source.

  Step 2: ROLE TEMPLATE
    If a template is selected, it is applied to ALL roles the user has after Step 1
    (plus any manually selected roles from the Basic section).
    Result: ModuleRolePermission entries for each role are overwritten by template values.
    UserModulePermission entries (cloned overrides) are NOT touched.

  Step 3: SPECIAL OVERRIDES
    If any overrides are explicitly set in the Special Overrides section,
    they are written to UserModulePermission.
    Result: User override values win over role/template values per RBAC precedence.

Special cases:
  - Clone + no template: user gets exact clone of source roles/overrides.
  - Template + no clone: user gets basic info + assigned roles + template-applied role permissions.
  - Clone + template + overrides: all three steps execute in order.
  - Neither clone nor template: user created with basic info + manually assigned roles + overrides.

  function store(Request $request)
  {
      // 1. Validate (existing fields + new fields)
      $validated = $request->validate([...]);

      DB::transaction(function () use ($request) {
          // 2. Create user (existing)
          $user = User::create([...]);

          // 3. Set suspended_at if status = suspended (existing pattern)
          if ($request->input('status') === 'suspended') {
              $user->suspended_at = now();
              $user->save();
          }

          // 4. Handle clone (Step 1)
          if ($request->filled('clone_user_id')) {
              $sourceUser = User::findOrFail($request->input('clone_user_id'));

              // Clone roles
              if ($request->boolean('copy_roles')) {
                  $roleIds = $sourceUser->roles->pluck('id')->toArray();
                  // Prevent cloning super-admin role unless confirmed
                  $this->preventSuperAdminAssignment($roleIds);
                  $user->roles()->syncWithoutDetaching($roleIds);
              }

              // Clone overrides
              if ($request->boolean('copy_overrides')) {
                  $overrides = UserModulePermission::where('user_id', $sourceUser->id)->get();
                  foreach ($overrides as $override) {
                      UserModulePermission::create([
                          'user_id' => $user->id,
                          'module_id' => $override->module_id,
                          ...permission columns from override
                      ]);
                  }
              }

              // Clone status
              if ($request->boolean('copy_status')) {
                  if ($sourceUser->suspended_at) {
                      $user->suspended_at = $sourceUser->suspended_at;
                      $user->save();
                  }
              }
          }

          // 5. Sync manually selected roles (existing)
          // Merge with cloned roles
          // Note: syncWithoutDetaching used to keep cloned roles
          if ($request->has('roles')) {
              $user->roles()->syncWithoutDetaching($request->input('roles', []));
          }

          // 6. Apply template (Step 2)
          if ($request->filled('apply_template')) {
              $template = RoleTemplate::where('slug', $request->input('apply_template'))->firstOrFail();

              // Dangerous check
              if ($template->is_dangerous && !$request->boolean('confirm_dangerous')) {
                  throw ValidationException::withMessages([
                      'confirm_dangerous' => 'You must confirm applying the Super Admin template.'
                  ]);
              }

              // Apply to all roles the user has
              $userRoles = $user->roles;
              $affectedModules = 0;
              foreach ($userRoles as $role) {
                  $affectedModules += $this->applyTemplateToRole($template, $role);
              }

              // Log activity for each role
              activity()
                  ->causedBy(Auth::user())
                  ->performedOn($user)
                  ->event('template_applied_during_creation')
                  ->withProperties([
                      'template' => ['id' => $template->id, 'name' => $template->name],
                      'role_count' => $userRoles->count(),
                      'affected_modules' => $affectedModules,
                  ])
                  ->log("Template '{$template->name}' applied to {$userRoles->count()} role(s) during user creation");
          }

          // 7. Handle special overrides (existing saveUserModulePermissions)
          // This is Step 3 — runs after clone and template
          // User override values always win over role/template
          $this->saveUserModulePermissions($user, $request);
      });

      return redirect()->route('users.index')
          ->with('success', 'User created successfully.');
  }

=== NEW HELPER: applyTemplateToRole() ===
  Extracted from RoleTemplateController::executeApply() to avoid duplication.

  Reusable by both RoleTemplateController and UserController:

  public function applyTemplateToRole(RoleTemplate $template, Role $role): int
  {
      $permissionsJson = $template->permissions_json;
      $moduleSlugs = array_keys($permissionsJson);
      $modules = Module::whereIn('slug', $moduleSlugs)->get()->keyBy('slug');
      $columns = RoleTemplate::$permissionColumns;
      $affectedCount = 0;

      foreach ($permissionsJson as $slug => $perms) {
          $module = $modules->get($slug);
          if (!$module) continue;

          $data = [];
          foreach ($columns as $col) {
              $data[$col] = (bool) ($perms[$col] ?? false);
          }

          ModuleRolePermission::updateOrCreate(
              ['module_id' => $module->id, 'role_id' => $role->id],
              $data
          );
          $affectedCount++;
      }

      return $affectedCount;
  }

  The RoleTemplateController::apply() method is refactored to also call this helper,
  so template application logic is shared, not duplicated.

============================================
HOW TEMPLATE APPLIES DURING USER CREATION
============================================

The template is applied to the user's role(s) AFTER the user exists and has roles.

1. User is created and saved to DB (has ID)
2. Roles are synced (from clone + manual selection)
3. Template's applyTemplateToRole() iterates each role and upserts
   ModuleRolePermission rows matching the template's permission JSON
4. Only WRITABLE columns are upserted (7 columns, excluding can_import
   which does not exist on module_role_permissions)
5. Template does NOT touch UserModulePermission at all
6. Activity is logged with event 'template_applied_during_creation'

If the user has no roles when a template is selected:
  - Validation prevents this: user must select at least one role
  - Error message: "You must assign at least one role to apply a template."

============================================
HOW CLONE AND TEMPLATE CONFLICT IS RESOLVED
============================================

Scenario: Both clone and template selected

Step 1 — Clone:
  - Source user's roles are attached to new user (syncWithoutDetaching)
  - Source user's overrides are copied (same module_id, same values)
  - Source user's status is copied (if checked)
  - Source user's password is NEVER copied

Step 2 — Roles synced:
  - Manually selected roles from Section 1 are also attached
  - No roles are detached (clone + manual = union)

Step 3 — Template applied:
  - Template upserts ModuleRolePermission for each role the user has
  - This OVERWRITES any role permissions that existed from clone source
  - User overrides (cloned in step 1) are NOT affected

Step 4 — Special overrides:
  - Written to UserModulePermission
  - These take precedence over template values (per RBAC rules)

Result: Cloned user overrides survive template application.
        Template overwrites role-level permissions.
        Special overrides win over everything.

============================================
SAFETY RULES
============================================

1. Super Admin only — entire route is under auth + role:super-admin middleware.
2. Password always required — never inherited from clone.
3. Super Admin template requires confirm_dangerous checkbox.
4. Super Admin role cannot be assigned via clone or manual role selection
   (existing preventSuperAdminAssignment logic).
5. Clone source user cannot be the newly created user (N/A for create, guard for future).
6. User overrides are optional and collapsed by default.
7. Dangerous permission warning — can_reveal/can_export/can_import detected in
   template summary and shown as warning.
8. Validation ensures at least one role is selected if template is applied.
9. Clone does not copy password, API tokens, sessions, MFA, or owned records
   (same constraint as existing clone implementation).
10. Transactional — entire create+clone+template+overrides wrapped in DB::transaction.

============================================
RISKS
============================================

1. Form Complexity
   Five sections on a single page could overwhelm some users.
   Mitigation: Sections 3 and 4 are collapsed by default; progressive disclosure.

2. JS Dependency
   Template summary, clone search, and review summary rely on JavaScript.
   Mitigation: degrade gracefully — template selection reloads page with query param
   to show summary server-side; clone search uses <select> fallback.

3. Race Condition — Template Applied to Wrong Role
   If clone and manual roles both assign roles, template applies to all of them.
   The user might not intend the template to apply to cloned roles.
   Mitigation: clear labels in UI showing "Template will be applied to ALL roles."

4. Template Apply Performance
   Applying template to many roles with many modules could be slow.
   Mitigation: always within DB transaction; batch operations via upsert.

5. User Override Confusion
   User might set overrides that contradict template values and not realize.
   Mitigation: summary shows override count; warning note when both are active.
