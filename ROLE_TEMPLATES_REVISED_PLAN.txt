ROLE TEMPLATES — REVISED PLAN (v2)
===================================
Date: 2026-06-25
Supersedes: ROLE_TEMPLATES_MASTER_PLAN.txt (v1)
Depends on: RBAC v1.0 (frozen) — 167 tests, 386 assertions

This document defines the Role Templates feature. Templates are pre-configured
permission blueprints that a Super Admin can apply to any role in one click.
They do NOT replace ModuleRolePermission — they only pre-fill it.

============================================
ARCHITECTURE RULE
============================================

Templates do NOT create a new permission layer. Applying a template copies
permission values into the existing ModuleRolePermission table. The RBAC
resolution logic (canOnModule, getEffectiveModulePermissions, etc.) is
completely unchanged. All existing RBAC v1.0 rules continue to apply.

User overrides (user_module_permissions) are NEVER touched by template
application — they continue to take precedence per RBAC rules.

============================================
DB CHANGES REQUIRED
============================================

=== Table: role_templates ===

  id              bigIncrements
  name            string(255)
  slug            string(255), unique
  description     text, nullable
  is_protected    boolean, default false
                    true = cannot be edited or deleted via UI
  is_dangerous    boolean, default false
                    true = template grants near-full access; apply
                    requires explicit confirmation checkbox
  created_at      timestamp
  updated_at      timestamp

=== Table: role_template_modules ===

  id              bigIncrements
  role_template_id  FK -> role_templates (cascade)
  module_id         FK -> modules (cascade)
  can_create      boolean, not null, default false
  can_read        boolean, not null, default false
  can_update      boolean, not null, default false
  can_delete      boolean, not null, default false
  can_approve     boolean, not null, default false
  can_export      boolean, not null, default false
  can_reveal      boolean, not null, default false
  can_import      boolean, not null, default false
  created_at      timestamp
  updated_at      timestamp

  UNIQUE: role_template_id, module_id

No changes to existing RBAC tables (module_role_permissions,
user_module_permissions, users, roles).

============================================
APPLY TEMPLATE TO ROLE — LOGIC
============================================

When a Super Admin applies a template to a role:

  1. Load all rows from role_template_modules for the template.
  2. Compute diff: compare current ModuleRolePermission values for
     the target role vs. what the template would set. Show user:
     - Modules that will be ADDED (no existing permission row)
     - Modules that will be OVERWRITTEN (existing row has different values)
     - Modules that are UNCHANGED (existing row matches template)
  3. User MUST confirm the overwrite via explicit checkbox/button.
  4. If template is is_dangerous = true (Super Admin template),
     require additional confirmation checkbox acknowledging the risk.
  5. Wrap in DB::transaction():
     a. For each module row in the template:
        UPSERT into module_role_permissions:
          role_id = target role's id
          module_id = template module's id
          All 8 permission columns = template values
     b. If any step fails → full rollback.
  6. Log activity (event: 'template_applied') with:
     - template name/id
     - target role name/id
     - user who applied it
  7. User overrides (user_module_permissions) are NEVER modified.
     Existing overrides continue to take precedence per RBAC rules.

RULE: Must show diff preview before applying.
RULE: Must confirm overwrite before applying.
RULE: Must never overwrite user_module_permissions.
RULE: Apply must be wrapped in a DB transaction.

Option — "Strict Apply" (future enhancement):
  If enabled, after step 5, DELETE any module_role_permissions rows
  for the role where the template has no entry. This ensures the
  role's permissions exactly match the template. Default: off.

============================================
SAFETY & SECURITY RULES
============================================

1. Non-super-admin users CANNOT access any template routes
   (controller middleware: super-admin only).

2. Protected templates (is_protected = true) CANNOT be edited or deleted
   via UI. Controller returns 403 for edit/update/destroy on protected
   templates.

3. Dangerous templates (is_dangerous = true) require additional
   confirmation checkbox before applying.

4. Super Admin template (is_dangerous = true, is_protected = true)
   should not be applied to normal roles. The UI must show a prominent
   warning and require explicit additional confirmation.

5. User overrides (user_module_permissions) are NEVER modified by
   template application — guaranteed by implementation (only touches
   module_role_permissions).

6. Applying a template ALWAYS logs activity (Spatie Activitylog)
   for audit trail.

7. The Apply operation is wrapped in a DB transaction — any failure
   causes full rollback.

8. Force delete is NEVER granted by any template — it remains
   super-admin only per RBAC v1.0 freeze rule.

============================================
RISKS
============================================

1. Template Drift
   New modules added after template creation are not included. When
   applied, the template won't touch those modules, leaving whatever
   permissions the role already has (or none).
   Mitigation: "Sync to current modules" button on template edit.

2. Overwriting Without Recovery
   Apply overwrites existing ModuleRolePermission rows. There is no
   undo. Mitigation: diff preview before apply; confirmation required;
   optional backup feature (save current role permissions to JSON log).

3. Protected Template Bypass
   is_protected is a DB flag, not an absolute guarantee. Anyone with
   direct DB access can change it. Mitigation: enforce in controller
   (abort 403 for edit/delete on protected) and in validation.

4. Dangerous Template Misuse
   Super Admin template grants full permissions. A well-intentioned
   Super Admin might apply it to a normal role without understanding
   the consequences. Mitigation: is_dangerous flag, prominent warnings,
   double confirmation, audit logging.

5. Permissions Not Reflecting Reality
   A template grants maximum possible permissions, but user overrides
   can restrict individual users. Super Admin must understand that
   applying a template to a role is not the same as applying it to
   every user with that role.
