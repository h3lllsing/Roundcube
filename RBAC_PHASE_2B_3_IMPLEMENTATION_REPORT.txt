What was implemented:

    endpoints (7 controllers).

3. GUARD PATTERN
--------------------------------------------------------------------------------

  Each reveal method received the identical 2-line guard inserted after
  userOwnedFilter() + findOrFail() and before any activity logging:

    $user = Auth::user();
    abort_unless($user->hasRole('super-admin') || ($record->module && $user->canOnModule($record->module, 'reveal')), 403);

  Where $record is the model instance (hosting, vps, voip, etc.).

  Flow:

    1. userOwnedFilter() — Phase 1 RbacScope query filter.
       - Super-admin: no scope. All records visible.
       - Non-super-admin: scoped to modules where user has can_read.
         Records with module_id = null are excluded → 404 for non-super-admin.
    2. findOrFail($id) — 404 if not found (or scoped out).
    3. Guard — abort_unless(super-admin or can_reveal), 403.
    4. Activity log — only reached if guard passes.
    5. Response — JSON (password/json) or Redirect (Vault).

  VaultController.reveal():
    Same guard pattern, but returns a redirect response with a session
    flash variable ('revealed_password') instead of JSON.

===============================================================================
4. PERMISSION RESOLUTION
--------------------------------------------------------------------------------

  All guards use the existing canOnModule() method from the
  HasModulePermissions trait, which resolves the effective permission:

    1. If a UserModulePermission override exists with a non-null value for
       can_reveal, return that value.
    2. Otherwise, return true if any of the user's roles have can_reveal
       set to true in module_role_permissions.

  Super-admin bypass: all guards check hasRole('super-admin') first, which
  short-circuits the permission check.

  No-module edge case:
    - $record->module is null → ($record->module && ...) evaluates to
      false → abort 403 for non-super-admin.
    - In practice, records with module_id = null are caught earlier by
      Phase 1 RbacScope (returning 404), so this is defense-in-depth.

8. SECURITY
--------------------------------------------------------------------------------

  - No privilege escalation introduced
      Guards only restrict, never grant. Users without can_reveal are
      blocked with 403.

  - User overrides are correctly handled
      Override=true can grant reveal even when role denies.
      Override=false can deny reveal even when role grants.

  - Denied reveals never create audit log entries
      Guard placed before activity() call, so denied requests never
      leave an audit trail. This prevents information leakage through
      log inspection.

  - Super-admin always bypasses
      No permission restriction applies to super-admin users.

  - Records without a module are inaccessible to non-super-admin
      Phase 1 RbacScope returns 404 before the guard fires.
      The guard's $record->module check is defense-in-depth.

  - No route changes
      All routes and URL structures unchanged.

  - No database changes
      All required columns already exist from Phase 2A/2B-1 migrations.

  - Backward compatible
      Users with appropriate can_reveal permission continue to reveal
      passwords as before. The new guards only add restrictions for
      users who lack the required permission.
